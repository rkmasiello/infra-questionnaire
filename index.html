Application

The app itself — accounting correctness, the property-management extension, and how tenants get onboarded. First, the parts that looked sound, so this reads as a map, not a teardown: money is NUMERIC(18,2) throughout (no float), journal entries enforce debits == credits at post time, and GL posting is idempotent (a unique posting-anchor blocks double-posts). The items below are spots that looked strange, asymmetric, or unbuilt — for each: real bug, intended, or roadmap?

Accounting correctness
8.1CHECKCan a bill with separately-stated (exclusive) tax actually post?
Read: the AP GL builder debits the line amounts and credits bill.total_amount (which includes tax) with no tax debit leg — while the invoice side does credit Sales Tax Payable. So DR (subtotal) ≠ CR (subtotal+tax) and the balance check should reject it. Missing leg, or is tax always baked into lines? gl-posting-engine.ts buildBillRows vs buildInvoiceRows
8.2CHECKIs there any period close / posting lock — or can entries be back-dated into already-reported months?
Read: no period-close / lock exists anywhere (0 matches across code + 160 migrations). Postings use the document's own entry_date with no comparison to a lock date. Intended for now, or on the roadmap?
8.3CHECKOver-application on payments — reject, clamp, or track as unapplied credit?
Read: the bill-payment apply path subtracts from bills.balance with no clamp (the reverse path clamps), so over-applying drives a bill negative; AR floors at 0 but silently swallows the excess. There's no persisted unapplied-cash / on-account credit. bill-payments.ts:459,771
Property-management logic
8.4CHECKIs rent actually "on the books," or only in the AR sub-ledger?
Read: rent / late-fee templates are auto_post=false and generated invoices stay draft; GL backfill only posts posted invoices. So rent shows in AR aging but the GL revenue + AR debit aren't recorded until someone posts each invoice. Is manual per-invoice posting the intended workflow?
8.5CHECKSecurity deposits & trust accounting — how are deposits kept out of revenue?
Read: portal receipts all post as ar_receipt (DR cash / CR AR), so a deposit reduces revenue-AR instead of crediting a deposit liability — and there's no trust / escrow / deposit-liability account type at all ("trust"/"escrow" appear only as labels). Trust accounting is a Buildium/AppFolio staple — on the roadmap?
8.6CHECKOwner accounting — is the management fee and the amount owed to owners actually booked?
Read: owner distributions book only DR clearing / CR cash — no "due to owner" payable is ever established or drawn down, and the management fee is computed but never posted (only netted inside the distribution). The owner statement also pulls cash from ar_receipts (by created_at) but opex from gl_entries (by entry_date) and nets by hand, so it won't tie to the ledger. Intended, or missing legs?
8.7CHECKWhich PM primitives are built vs. roadmap — proration, NSF/returned payments, deposit disposition, owner 1099?
Read: rent always bills the full month (no move-in/out proration); no NSF/ACH-return flow; security-deposit disposition is a documented stub; owner 1099 is vendor-only (re_owners.requires_1099 is collected but consumed by nothing). Roadmap or oversight?
Product, tenancy & onboarding
8.8How is a new tenant provisioned today, and how automated?
Read: a multi-step admin flow — POST /tenants/create → /onboard (ensureTenantTables) → seed-coa-all-tenants → seed-rbac → assign-user-to-tenants. Scripted end-to-end, or hand-run per tenant?
8.9Is there self-serve signup, an admin console to manage tenants, or is it API + scripts + SQL?
Read: onboarding is invite-token based (invitations.ts) — no self-serve signup. There's a platform-admin concept (grant-platform-admin CLI, platform-admin routes) but no full tenant-management UI surfaced.
8.10How do tenants pay you — in-product subscription / metering, or off-platform? Any plan tiers / entitlements?
Read: no SaaS billing in the code — the invoicing/AR is the ERP charging its customers, not you charging tenants; Stripe appears only for RE resident payments. The only per-tenant metering is AI usage budgets (migration 155). So billing is off-platform today?
8.11Tenant lifecycle — can you suspend, offboard, export, or hard-delete a single tenant?
Read: dropTenantSchema works only in schema mode; flat mode (prod) has no clean per-tenant delete, and the only wipe is a dev-only __dev__/reset-all. Looks unbuilt — confirm.
